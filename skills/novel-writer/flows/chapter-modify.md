# 章节修改流程

> 本文件包含：章节重写、段落重写、手工编辑
> 归档命名：`第XXX章-章节名--YYYYMMDD-HHMMSS--操作类型.md`（操作类型：rewrite / paragraph / pre-edit）

---

## 章节重写

触发：`重写`、`重写第X章`、`推翻重来`、`这章不行`、`重新写`

对已写章节进行大幅度重写——推翻原有内容按新方向重新创作（区别于润色：润色保留剧情结构仅优化文风）。

**流程：**

1. **确认重写目标**：
   - 用户指定章节编号（支持单章或连续多章，如"重写第3-5章"）
   - 如果未指定，默认为最新一章
2. **加载上下文**（与智能续写相同的分层加载）：
   - 读取 PROJECT.yaml
   - 读取最近章节日志：`bash ${CLAUDE_SKILL_DIR}/scripts/tail-log.sh <项目目录>/chapter-log.md 5`（**严禁直接 Read 整个 chapter-log.md**，与写作流程规则一致）
   - 读取 outline/大纲.md（当前章节段落）
   - 读取目标章节原文
   - 读取前后章节概况（确保衔接）
3. **分析原文问题**：
   - 展示原文概况和当前大纲对比
   - 询问用户重写原因和新方向：
   ```
   《书名》第X章「章节名」重写准备
   ━━━━━━━━━━━━━━━━━━━━
   原文概况：（chapter-log 中的记录）
   大纲要求：（大纲中本章的计划）
   原文字数：约XXXX字
   ━━━━━━━━━━━━━━━━━━━━
   请描述重写方向（如：改变剧情走向、更换视角、
   加强冲突等），或直接说"按大纲重写"。
   ```
4. **备份原文**：使用脚本归档
   ```bash
   node ${CLAUDE_SKILL_DIR}/scripts/archive.js <章节文件> rewrite <chapters_dir>
   ```
5. **重写章节**：按用户指定的新方向重写，遵循写作原则（参见 `Read ${CLAUDE_SKILL_DIR}/flows/writing.md` 中的"写作"章节）
6. **保存（内联执行）**：
   - 更新 chapter-log.md（替换原条目，标注"重写"）
   - 更新 PROJECT.yaml：`current_chapter`、`updated`、`active_characters`、`focus_plotlines`；`status` 流转规则同写作流程；`last_action: {type: "rewrite", target: "第XXX章", timestamp: 当前时间}`
   - 按需更新角色档案（P1）、伏笔表（P1）
7. **检查前后衔接**：
   - 自动检查重写内容与前一章结尾、后一章开头是否衔接
   - 如有断裂，提示用户并建议修改方案

**展示格式：**

```
第X章「章节名」重写完成 ✓
━━━━━━━━━━━━━━━━━━━━
原文归档：chapters/_history/第XXX章-章节名--YYYYMMDD-HHMMSS--rewrite.md
新文字数：约XXXX字（原文约XXXX字）
━━━━━━━━━━━━━━━━━━━━
变更摘要：
  - 剧情变化：...
  - 伏笔影响：新增X条 / 移除X条
  - 衔接状态：与前后章节衔接正常 / 需注意后一章开头
━━━━━━━━━━━━━━━━━━━━
```

## 段落重写（精细重写）

触发：`重写段落`、`改这段`、`这段重写`、`精修`、`重写我选中的段落`、`这几段不好`、`局部重写`

对已写章节中的**指定段落**进行精细重写——不改动整章，只精准替换目标段落。归档类型 `paragraph`，保存流程为轻量执行。

**流程：**

1. 确认重写目标：
   - 用户指定章节编号（如"第3章的那段打斗重写一下"）
   - 如果未指定，默认为最新一章
2. 加载章节内容：
   - 读取 PROJECT.yaml
   - 读取最近章节日志：`bash ${CLAUDE_SKILL_DIR}/scripts/tail-log.sh <项目目录>/chapter-log.md 5`（**严禁直接 Read 整个 chapter-log.md**）
   - 读取目标章节全文
   - 读取前后章节概况（确保衔接不会被破坏）
3. 定位目标段落：
   - **禁止输出整章的带编号段落**（会造成严重刷屏灾难）。
   - 用户通过以下方式指定段落：
     - **直接摘录**：用户直接粘贴要重写的文本片段、包含几个关键词的短句。
     - **内容引用**：如"那段描写月色的"、"林清嘉说话那段"。
   - 如果用户描述模糊，大模型主动从文中提取 1-3 个最匹配的段落片段（只输出每段的前 10 个字和后 10 个字作为预览），供用户确认，例如：
   ```
   《书名》第X章「章节名」段落定位
   ━━━━━━━━━━━━━━━━━━━━
   找到以下可能匹配的段落：
   [A] 月光透过纱帘洒在案几上...（中略）...心中却久久难以平静。
   [B] 沈宛抬头看他，目光中闪过...（中略）...你难道真的不明白吗？
   ━━━━━━━━━━━━━━━━━━━━
   请确认是哪一段？（如：A、B）或者粘贴原文。
   ```
4. **确认重写方向**：
   - 展示选中段落的完整原文
   - 询问重写要求：
   ```
   已选中段落（约200字）：
   ━━━━━━━━━━━━━━━━━━━━
   （原文内容...）
   ━━━━━━━━━━━━━━━━━━━━
   上文衔接：...（前一段落末尾）
   下文衔接：...（后一段落开头）
   ━━━━━━━━━━━━━━━━━━━━
   请描述重写方向（如：加强紧张感、改为内心独白、
   补充环境描写、缩短节奏等），或说"你来优化"由我判断。
   ```
5. **备份并替换**：使用脚本化三锚点替换（自动归档+替换+校验一步完成）
   ```bash
   # 将新段落内容写入临时文件，然后调用 replace-paragraph.js
   node ${CLAUDE_SKILL_DIR}/scripts/replace-paragraph.js <章节文件> <新段落文件> \
     --before "<前文锚点>" --target "<要替换的原文>" --after "<后文锚点>"
   ```
   > 脚本自动执行：归档旧版本（类型 `rewrite-paragraph`）→ 三锚点定位（必须恰好 1 个匹配）→ 原位替换 → 长度校验。
   > 如果匹配 0 个：要求重新提供锚点。如果匹配 >1 个：要求补更长锚点。
   > 如果 `_history/` 中已有同一章 1 小时内的 `rewrite-paragraph` 备份，可跳过重复备份以节省空间。当同一章的 `_history/` 备份超过 10 个时，提示用户"历史版本较多，建议清理旧版本"，但不自动删除
6. **重写段落**：
   - 按用户指定的方向重写目标段落
   - **严格保持与前后段落的衔接**：开头承接上一段落末尾，结尾连接下一段落开头
   - 遵循写作原则（开头有冲突/悬念、每千字至少一个爽点、对话推进剧情、场景服务情绪、章末设钩子），保持与章节整体风格一致
   - 如果用户选了多个段落，可以合并/拆分段落，但需说明
   - 将重写后的段落写入临时文件，调用步骤 5 的 `replace-paragraph.js` 完成替换
   - 脚本会自动处理：归档、定位、替换、校验。如果脚本报告匹配失败，根据错误码调整锚点后重试
7. **轻量保存**：
   - chapter-log.md：在 `**字数**` 行后追加 `- **段落重写**：第N段, 方向: xxx, YYYY-MM-DD`，并重新统计字数
   - PROJECT.yaml：`updated`；`last_action: {type: "paragraph", target: "第XXX章", timestamp: 当前时间}`
   - 涉及角色行为变化 → 按需更新角色档案；涉及伏笔变化 → 按需更新 foreshadowing.md

**展示格式：**

```
第X章「章节名」段落重写完成 ✓
━━━━━━━━━━━━━━━━━━━━
原文归档：chapters/_history/第XXX章-章节名--YYYYMMDD-HHMMSS--rewrite-paragraph.md
重写段落：[A]（原约200字 → 新约280字）
━━━━━━━━━━━━━━━━━━━━
重写前：
  "沈宛抬头看他，目光中闪过一丝慌张..."
重写后：
  "沈宛猛地合上手中的古籍，指尖微微发颤..."
━━━━━━━━━━━━━━━━━━━━
衔接检查：与前后段落衔接正常 ✓
━━━━━━━━━━━━━━━━━━━━
```

**多段落重写时的展示格式：**

```
第X章「章节名」段落重写完成 ✓
━━━━━━━━━━━━━━━━━━━━
原文归档：chapters/_history/第XXX章-章节名--YYYYMMDD-HHMMSS--rewrite-paragraph.md
重写段落：[B]-[D]（原约600字 → 新约750字）
━━━━━━━━━━━━━━━━━━━━
变更摘要：
  - [B] 环境描写增强，渲染紧张氛围
  - [C]-[D] 合并为一段，加快对话节奏
衔接检查：与前后段落衔接正常 ✓
━━━━━━━━━━━━━━━━━━━━
```

## 手工编辑

触发：`编辑`、`手工编辑`、`改正文`、`我要自己改`、`打开编辑`、`自己改`

用户希望直接用文本编辑器修改章节正文，修改完成后系统自动同步。**极致懒人流程**：一句话打开文件，改完一句话同步。

**与其他修改方式的区别：**

| | 段落重写 | 章节重写 | 手工编辑 |
|---|---|---|---|
| 修改者 | AI | AI | 用户自己 |
| 粒度 | 指定段落 | 整章 | 用户自由决定 |
| 归档 | 替换时自动 | 替换时自动 | 打开时立即归档 |
| 适用场景 | 需要AI重写某段 | 整章方向错了 | 微调措辞、修改细节、自由修改 |

**流程分两步：**

### 第一步：打开编辑

1. **确认编辑目标**：
   - 用户指定章节编号（如「编辑第3章」）
   - 如果未指定，默认为 `current_chapter` 或最新一章
2. **调用脚本准备编辑环境**：
   ```bash
   node ${CLAUDE_SKILL_DIR}/scripts/edit-chapter.js <项目目录> <章节编号> --editor <编辑器命令>
   ```
   - `--editor` 参数传入 `workspace.yaml` 的 `editor` 字段值（如 `wps`、`code` 等）；若为空可省略
   - 脚本自动执行：规范化文本 → 归档当前版本（操作类型 `pre-edit`）→ 创建编辑标记（`.edit-marker-<编号>.json`）→ 如果检测到富文本编辑器（WPS/Word 等），自动将 md 转为 .docx → 返回 `open_path`（实际应打开的文件路径）
   - 返回 JSON 中 `edit_format` 字段为 `"md"` 或 `"docx"`，`open_path` 为应传给编辑器的文件
3. **打开文件编辑器**：
   - 先读取 `${CLAUDE_SKILL_DIR}/workspace.yaml` 的 `editor` 字段
   - **使用脚本返回的 `open_path`**（而非章节原始 .md 路径）作为编辑器打开的文件
   - **如果 `editor` 已配置**（非空）：`<editor> "<open_path>"`
     - 纯文本编辑器示例：`code "D:/novel/星渊坠落/chapters/第003章-风起.md"`
     - 富文本编辑器示例：`wps "D:/novel/星渊坠落/chapters/第003章-风起.edit.docx"`
   - **如果 `editor` 为空或未配置**，使用系统默认程序（跨平台）：
     - Windows: `start "" "<章节文件路径>"`
     - macOS: `open "<章节文件路径>"`
     - Linux: `xdg-open "<章节文件路径>"`
   - 如果打开失败，展示文件完整路径，提示用户手动打开
4. **展示提示**：
   ```
   第X章「章节名」已在编辑器中打开 ✓
   ━━━━━━━━━━━━━━━━━━━━
   文件位置：<完整路径>
   原文归档：chapters/_history/第XXX章-章节名--XXXXXXXX-XXXXXX--pre-edit.md
   当前字数：约XXXX字
   ━━━━━━━━━━━━━━━━━━━━
   编辑完成后回来说「编辑完了」或「同步」即可。
   系统会把同步视为原子操作：成功才提交；若不能完整同步，会自动回撤到编辑前版本。只要检测到可保留的改动内容，也会自动保存成草稿快照，避免你白写。
   ━━━━━━━━━━━━━━━━━━━━
   ```

### 第二步：编辑同步

触发：`编辑完了`、`同步`、`改好了`、`改完了`、`同步编辑`、`编辑同步`

1. **调用同步脚本**：
   ```bash
   node ${CLAUDE_SKILL_DIR}/scripts/sync-edit.js <项目目录> [章节编号]
   ```
   脚本自动执行：检测编辑标记中的 `edit_format` → 如果为 `docx` 则自动将 .docx 转回 .md（docx→md） → 检测变更（哈希比对）→ 规范化文本（CRLF→LF）→ 尝试完成必需同步步骤 → 仅在完整成功时提交；否则自动回撤到 `pre-edit` 并保存草稿快照
2. **如果无变更**：
   ```
   第X章「章节名」未检测到变更
   ━━━━━━━━━━━━━━━━━━━━
   编辑标记已清理。如果确实修改了文件，请确认已保存后重试。
   ━━━━━━━━━━━━━━━━━━━━
   ```
3. **如果完整成功**：
   ```
   第X章「章节名」手工编辑已同步 ✓
   ━━━━━━━━━━━━━━━━━━━━
   变更统计：原约XXXX字 → 现约XXXX字
   已更新：正文、章节日志、项目进度
   编辑标记已清理
   ━━━━━━━━━━━━━━━━━━━━
   建议操作：（按「智能建议引擎」生成）
   ━━━━━━━━━━━━━━━━━━━━
   ```
4. **如果不能完整同步**：
   ```
   第X章「章节名」同步未完成，已自动恢复到编辑前版本
   ━━━━━━━━━━━━━━━━━━━━
   本次手工修改未正式提交
   已保留草稿快照：chapters/_drafts/第XXX章-章节名--XXXXXXXX-XXXXXX--manual-edit-draft.md
   你可以稍后基于草稿继续处理
   ━━━━━━━━━━━━━━━━━━━━
   ```
   - 注：并非所有失败都一定能生成草稿快照；只有检测到可安全保留的正文改动内容时，系统才会自动落草稿
5. **失败判定原则**：
   - 只要必需同步步骤不能完整完成，就视为失败
   - 第一版的必需同步范围至少包括：正文文件可正常回收、项目状态可回写、章节日志可进入允许的同步路径
   - 失败时不得留下“正文已改但元数据未改”的半同步状态

**注意事项：**
- 编辑标记持久在磁盘，跨会话有效。说「同步」即可继续
- 打开同一章重新归档并覆盖旧标记；编辑标记不影响其他操作
- 支持同时编辑多章（每章独立标记），`sync-edit.js` 不带编号时一次同步全部
- 撤销：从 `_history/` 恢复 `pre-edit` 快照
- **草稿快照**：如果同步失败，用户改过的正文会落在 `chapters/_drafts/`，避免辛苦写的内容丢失
- **传递内容**：纯文本编辑器打开 .md 原文件；富文本编辑器（WPS/Word）自动生成 `.edit.docx`。两种模式用户都只改正文，系统负责决定提交还是回撤
- **WPS/Word 须知**：配置 `editor` 为 `wps`/`winword` 后自动转换，用户无需手动操作。同步前**必须先关闭编辑器**。勿插入图片/表格（转回 md 时丢失）
- **系统默认程序局限**：`editor` 留空时不会生成 .docx（无法检测默认程序类型），WPS/Word 用户必须显式配置
- **首次提醒**：若 `editor` 为空且 `workspace.yaml` 无 `editor_prompted: true`，提示一次配置编辑器，然后写入 `editor_prompted: true`

**编辑器兼容性说明（用户询问时提供）：**

| 编辑器 | 配置值 | 说明 |
|--------|--------|------|
| VS Code | `code` | 推荐。支持 Markdown 预览、语法高亮 |
| Notepad++ | `notepad++` | 轻量好用，需确认已加入系统 PATH |
| Sublime Text | `subl` | 需安装 subl 命令行工具 |
| 记事本 | `notepad` | 可用但功能极简，不推荐长篇编辑 |
| WPS 文字 | `wps` | 完整支持。系统自动将 .md 转为 .docx 供编辑，同步时自动转回 .md。用户无感知 |
| Word | `winword` | 完整支持。同 WPS，自动 md↔docx 双向转换 |
| LibreOffice | `libreoffice` | 完整支持。同 WPS/Word |
| Vim/Neovim | `vim` / `nvim` | 终端编辑器，会阻塞当前终端直到退出 |
| 系统默认 | 留空 | 使用操作系统关联的 .md 默认程序。注意：如果系统默认 .md 关联为 WPS/Word，应显式配置编辑器值以启用自动转换 |
