# 章节修改流程

> 本文件包含：章节重写、段落重写
> 归档命名：`第XXX章-章节名--YYYYMMDD-HHMMSS--操作类型.md`（操作类型：rewrite / paragraph）

---

## 章节重写

触发：`重写`、`重写第X章`、`推翻重来`、`这章不行`、`重新写`

对已写章节进行大幅度重写——不是润色微调，而是推翻原有内容，按新方向重新创作。

**与润色的区别：**

| | 润色 | 重写 |
|---|------------|------------|
| 目的 | 文风调整、语言优化 | 剧情方向、结构推翻重来 |
| 改动幅度 | 保留原有剧情和结构 | 可能完全改变剧情走向 |
| 触发场景 | 写完觉得文笔不够好 | 写完觉得情节走歪了 |

**流程：**

1. **确认重写目标**：
   - 用户指定章节编号（支持单章或连续多章，如"重写第3-5章"）
   - 如果未指定，默认为最新一章
2. **加载上下文**（与智能续写相同的分层加载）：
   - 读取 PROJECT.yaml
   - 读取最近章节日志：`bash ${CLAUDE_SKILL_DIR}/scripts/tail-log.sh <项目目录>/chapter-log.md 5`（**严禁直接 Read 整个 chapter-log.md**，与写作流程规则一致）
   - 读取 outline/大纲.md（当前章节段落）
   - 读取目标章节原文
   - 读取前后章节概况（确保衔接）
3. **分析原文问题**：
   - 展示原文概况和当前大纲对比
   - 询问用户重写原因和新方向：
   ```
   《书名》第X章「章节名」重写准备
   ━━━━━━━━━━━━━━━━━━━━
   原文概况：（chapter-log 中的记录）
   大纲要求：（大纲中本章的计划）
   原文字数：约XXXX字
   ━━━━━━━━━━━━━━━━━━━━
   请描述重写方向（如：改变剧情走向、更换视角、
   加强冲突等），或直接说"按大纲重写"。
   ```
4. **备份原文**：使用脚本归档
   ```bash
   node ${CLAUDE_SKILL_DIR}/scripts/archive.js <章节文件> rewrite <chapters_dir>
   ```
5. **重写章节**：按用户指定的新方向重写，遵循写作原则（参见 `Read ${CLAUDE_SKILL_DIR}/flows/writing.md` 中的"写作"章节）
6. **保存（内联执行，无需额外读取 writing.md）**：
   - 更新 chapter-log.md（替换原条目，标注"重写"），格式：`## 第X章 - [章节标题]` + 概况/关键事件/人物变化/伏笔/字数
   - 更新 PROJECT.yaml：`current_chapter`、`updated`、`active_characters`、`focus_plotlines`；`status` 流转规则同写作流程；`last_action: {type: "rewrite", target: "第XXX章", timestamp: 当前时间}`。如需兼容旧工作流，可选择性维护 `workspace.yaml.active_project`，但它不应覆盖当前会话里已明确的项目
   - 按需更新角色档案（P1）、伏笔表（P1）——重写可能改变伏笔状态
7. **检查前后衔接**：
   - 自动检查重写内容与前一章结尾、后一章开头是否衔接
   - 如有断裂，提示用户并建议修改方案

**展示格式：**

```
第X章「章节名」重写完成 ✓
━━━━━━━━━━━━━━━━━━━━
原文归档：chapters/_history/第XXX章-章节名--YYYYMMDD-HHMMSS--rewrite.md
新文字数：约XXXX字（原文约XXXX字）
━━━━━━━━━━━━━━━━━━━━
变更摘要：
  - 剧情变化：...
  - 伏笔影响：新增X条 / 移除X条
  - 衔接状态：与前后章节衔接正常 / 需注意后一章开头
━━━━━━━━━━━━━━━━━━━━
```

## 段落重写（精细重写）

触发：`重写段落`、`改这段`、`这段重写`、`精修`、`重写我选中的段落`、`这几段不好`、`局部重写`

对已写章节中的**指定段落**进行精细重写——不改动整章，只精准替换目标段落，保留上下文衔接。

**与章节重写的区别：**

| | 章节重写 | 段落重写 |
|---|------------|--------------|
| 粒度 | 整章推翻重来 | 仅替换指定段落 |
| 改动范围 | 可能完全改变剧情走向 | 保留整体剧情，局部优化 |
| 触发场景 | 整章方向错了 | 某几段写得不满意、节奏不对、描写不到位 |
| 归档类型 | `rewrite` | `paragraph` |
| 保存流程 | 完整执行 | 轻量执行（仅更新 chapter-log 标注"段落重写"） |

**流程：**

1. 确认重写目标：
   - 用户指定章节编号（如"第3章的那段打斗重写一下"）
   - 如果未指定，默认为最新一章
2. 加载章节内容：
   - 读取 PROJECT.yaml
   - 读取最近章节日志：`bash ${CLAUDE_SKILL_DIR}/scripts/tail-log.sh <项目目录>/chapter-log.md 5`（**严禁直接 Read 整个 chapter-log.md**）
   - 读取目标章节全文
   - 读取前后章节概况（确保衔接不会被破坏）
3. 定位目标段落：
   - **禁止输出整章的带编号段落**（会造成严重刷屏灾难）。
   - 用户通过以下方式指定段落：
     - **直接摘录**：用户直接粘贴要重写的文本片段、包含几个关键词的短句。
     - **内容引用**：如"那段描写月色的"、"林清嘉说话那段"。
   - 如果用户描述模糊，大模型主动从文中提取 1-3 个最匹配的段落片段（只输出每段的前 10 个字和后 10 个字作为预览），供用户确认，例如：
   ```
   《书名》第X章「章节名」段落定位
   ━━━━━━━━━━━━━━━━━━━━
   找到以下可能匹配的段落：
   [A] 月光透过纱帘洒在案几上...（中略）...心中却久久难以平静。
   [B] 沈宛抬头看他，目光中闪过...（中略）...你难道真的不明白吗？
   ━━━━━━━━━━━━━━━━━━━━
   请确认是哪一段？（如：A、B）或者粘贴原文。
   ```
4. **确认重写方向**：
   - 展示选中段落的完整原文
   - 询问重写要求：
   ```
   已选中段落（约200字）：
   ━━━━━━━━━━━━━━━━━━━━
   （原文内容...）
   ━━━━━━━━━━━━━━━━━━━━
   上文衔接：...（前一段落末尾）
   下文衔接：...（后一段落开头）
   ━━━━━━━━━━━━━━━━━━━━
   请描述重写方向（如：加强紧张感、改为内心独白、
   补充环境描写、缩短节奏等），或说"你来优化"由我判断。
   ```
5. **备份并替换**：使用脚本化三锚点替换（自动归档+替换+校验一步完成）
   ```bash
   # 将新段落内容写入临时文件，然后调用 replace-paragraph.js
   node ${CLAUDE_SKILL_DIR}/scripts/replace-paragraph.js <章节文件> <新段落文件> \
     --before "<前文锚点>" --target "<要替换的原文>" --after "<后文锚点>"
   ```
   > 脚本自动执行：归档旧版本（类型 `rewrite-paragraph`）→ 三锚点定位（必须恰好 1 个匹配）→ 原位替换 → 长度校验。
   > 如果匹配 0 个：要求重新提供锚点。如果匹配 >1 个：要求补更长锚点。
   > 如果 `_history/` 中已有同一章 1 小时内的 `rewrite-paragraph` 备份，可跳过重复备份以节省空间。当同一章的 `_history/` 备份超过 10 个时，提示用户"历史版本较多，建议清理旧版本"，但不自动删除
6. **重写段落**：
   - 按用户指定的方向重写目标段落
   - **严格保持与前后段落的衔接**：开头承接上一段落末尾，结尾连接下一段落开头
   - 遵循写作原则（开头有冲突/悬念、每千字至少一个爽点、对话推进剧情、场景服务情绪、章末设钩子），保持与章节整体风格一致
   - 如果用户选了多个段落，可以合并/拆分段落，但需说明
   - 将重写后的段落写入临时文件，调用步骤 5 的 `replace-paragraph.js` 完成替换
   - 脚本会自动处理：归档、定位、替换、校验。如果脚本报告匹配失败，根据错误码调整锚点后重试
7. **轻量保存**：
   - 更新 chapter-log.md：在对应章节条目的 `**字数**` 行之后追加一行 `- **段落重写**：第N段, 方向: xxx, YYYY-MM-DD`，并重新统计章节字数更新 `**字数**` 行（使用 `wc -m` 或 Read 工具统计替换后的实际字数）
   - 更新 `PROJECT.yaml`：`updated` 字段；`last_action` 更新为 `{type: "paragraph", target: "第XXX章", timestamp: 当前时间}`。如需兼容旧工作流，可选择性维护 `workspace.yaml.active_project`，但它不应覆盖当前会话里已明确的项目
   - 如果段落重写涉及角色行为变化 → 按需更新角色档案
   - 如果段落重写涉及伏笔变化 → 按需更新 foreshadowing.md
   - 不重新生成完整章节日志（保留原有条目）

**展示格式：**

```
第X章「章节名」段落重写完成 ✓
━━━━━━━━━━━━━━━━━━━━
原文归档：chapters/_history/第XXX章-章节名--YYYYMMDD-HHMMSS--rewrite-paragraph.md
重写段落：[A]（原约200字 → 新约280字）
━━━━━━━━━━━━━━━━━━━━
重写前：
  "沈宛抬头看他，目光中闪过一丝慌张..."
重写后：
  "沈宛猛地合上手中的古籍，指尖微微发颤..."
━━━━━━━━━━━━━━━━━━━━
衔接检查：与前后段落衔接正常 ✓
━━━━━━━━━━━━━━━━━━━━
```

**多段落重写时的展示格式：**

```
第X章「章节名」段落重写完成 ✓
━━━━━━━━━━━━━━━━━━━━
原文归档：chapters/_history/第XXX章-章节名--YYYYMMDD-HHMMSS--rewrite-paragraph.md
重写段落：[B]-[D]（原约600字 → 新约750字）
━━━━━━━━━━━━━━━━━━━━
变更摘要：
  - [B] 环境描写增强，渲染紧张氛围
  - [C]-[D] 合并为一段，加快对话节奏
衔接检查：与前后段落衔接正常 ✓
━━━━━━━━━━━━━━━━━━━━
```
