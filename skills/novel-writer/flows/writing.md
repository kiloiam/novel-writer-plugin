# 写作推进流程

> 本文件包含：智能续写、写作、章节保存流程、前情回顾、灵感激发

---

## 智能上下文管理

上下文窗口有限，不可能每次全读所有文件。写作相关操作按以下分层策略加载：

| 层级 | 何时加载 | 文件 |
|------|---------|------|
| L0 索引 | 每次操作 | PROJECT.yaml |
| L1 近况 | 写作/续写 | **严禁直接 Read 整个 chapter-log.md**，必须使用脚本：`node ${CLAUDE_SKILL_DIR}/scripts/tail-log.js <项目目录>/chapter-log.md 5` |
| L2 计划 | 写作/续写 | outline/大纲.md（当前章节段落） |
| L3 角色 | 按需 | characters/[active_characters].md（文件不存在则忽略） |
| L4 设定 | 按需 | worldbuilding/[与本章相关].md |
| L5 伏笔 | 快速扫描 | foreshadowing.md |

**判断"相关"的方法**：先看 PROJECT.yaml 的 `active_characters` 和 `focus_plotlines`，再看大纲中本章细纲提到的角色和场景。如有不确定，宁可多读一两个文件。

> `active_characters` / `focus_plotlines` 是**有上限的滚动工作集**，不是“只根据本章即时重算”的瞬时结果。加载时应把它们理解为“最近几章仍值得持续关注的核心角色/情节线”，而不是“本章未出现就可以直接踢出”。

## 智能续写

触发：`继续`、`继续写`、`下一章`、`写`、`开始写`、`接着写`、`写一下`

这是最高频的操作，流程设计优先体验：

**默认规则：**
- 用户未指定项目时，默认使用当前或最近活跃项目
- 用户未指定章节时，默认继续当前章节、最近编辑章节，或下一章计划章节
- **0 章项目**：如果项目还没有任何章节（现存 0 章），自动从第 1 章开始写，无需追问
- 只有在没有明显目标章节时，才追问"要写哪一章"

**上下文加载**：按第五节「智能上下文管理」的分层策略加载（L0-L5）。

**加载完毕后，向用户展示简报：**

```
《书名》续写准备就绪
━━━━━━━━━━━━━━━━━━━━
进度：当前第6章 / 计划共20章 / 现存6章（第一卷）
上章：「章节名」— 概况摘要
本章：「章节名」— 大纲要点
活跃角色：A、B、C
待揭伏笔：2条
特别备注：（next_chapter_note 内容）
━━━━━━━━━━━━━━━━━━━━
是否开始写作？有什么特别要求？
```

用户确认后进入写作，写完后执行 **章节保存流程**。

## 写作

触发：`写第X章`、`推进`、`扩写`、`演绎`

将章节细纲扩写为完整正文。

**写作原则：**
- 开头 3 段必须有冲突或悬念
- 每 1000 字至少一个小爽点
- 对话推进剧情，避免无效寒暄
- 场景描写服务于情绪
- 世界观细节自然融入，不强行解释
- 章末设置钩子

**写完后自动执行章节保存流程。**

**参考资料**（按需读取，不要每次都读）：
- 写作技巧：`Read ${CLAUDE_SKILL_DIR}/references/writing-tips.md`
- 避坑指南：`Read ${CLAUDE_SKILL_DIR}/references/common-pitfalls.md`

## 前情回顾

触发：`回顾`、`前情`、`之前写了什么`

不写新内容，只生成前文摘要：

1. 先检查 `chapter-log.md` 行数：`wc -l < <项目目录>/chapter-log.md`
   - **≤ 500 行**：可以直接读取全文
   - **> 500 行**：使用 `node ${CLAUDE_SKILL_DIR}/scripts/tail-log.js <log_file> 20` 读取最近 20 章替代
2. 读取 `foreshadowing.md` 中的活跃伏笔
3. 输出结构化摘要：各章概况 → 人物状态 → 未揭伏笔 → 当前情节线

适用于长时间中断后恢复写作。

## 章节保存流程（写完后自动执行）

每次写完一章后，**推荐使用网关脚本一次性完成所有 P0 操作**：

```bash
# LLM 先将正文写入临时文件，再调用网关：
node ${CLAUDE_SKILL_DIR}/scripts/save-chapter.js <项目目录> <章节编号> <正文临时文件> \
  --title "章节标题" \
  --log-entry <日志条目临时文件> \
  --characters "角色A,角色B" \
  --plotlines "主线:XXX,支线:YYY" \
  --clear-note
```

脚本自动执行：覆盖保护（归档旧版）→ 净化文件名 → 保存正文 → 统计字数 → 追加日志 → 更新 PROJECT.yaml。
LLM 只需负责创作正文和生成日志条目内容。

**手动流程（网关脚本不可用时的 fallback）：**

**P0 必做（每次都执行）：**
1. **覆盖保护**：如果写入目标章节文件已存在（即覆盖旧章节而非新建），必须先归档旧正文：
   ```bash
   node ${CLAUDE_SKILL_DIR}/scripts/archive.js <章节文件> write <chapters_dir>
   ```
   纯新建章节（目标文件不存在）无需此步骤
2. 保存正文 → `chapters/第XXX章-[章节名].md`
2. 追加章节日志 → `chapter-log.md`，使用以下格式：
   ```markdown
   ## 第X章 - [章节标题]
   
   - **概况**：一两句话概括本章核心内容
   - **关键事件**：
     - 事件1
     - 事件2
   - **人物变化**：涉及的角色状态/关系变化
   - **伏笔**：本章埋设或揭示的伏笔（同步更新 foreshadowing.md）
   - **字数**：约XXXX字
   ```
   > 概况要简洁但足以回忆剧情走向；人物变化记录等级提升、关系转变、性格转折、获得道具等
3. 更新进度 → `PROJECT.yaml`：**推荐使用脚本**以保证结构完整性：
   ```bash
   node ${CLAUDE_SKILL_DIR}/scripts/update-project.js <项目目录> \
     --chapter <N> --status writing \
     --add-character "新角色" \
     --last-action '{"type":"write","target":"第XXX章","timestamp":"当前时间"}'
   ```
   手动更新仅作 fallback。字段规则：
   - `current_chapter` 更新为当前最新章节编号（写作指针）
   - `total_chapters` **不在此步骤修改**——它仅代表计划总章数，只有用户主动调整计划或大纲变更时才更新
   - `status` 自动流转：如果当前为 `planning` 则更新为 `writing`；如果当前为 `polishing` 且正在写新章节（非润色后保存），则回退为 `writing`
   - `active_characters` 不是按“仅本章出场角色”彻底重算，而是以旧值为基础保留近期仍重要的核心角色，再合并本章新增核心角色；建议控制在 3-6 个，避免角色只因单章缺席就从后续加载入口消失
   - `focus_plotlines` 同样使用滚动保留策略：保留近期仍在推进的主/支线，再合并本章新增情节线；建议控制在 2-4 条，避免状态页落后于正文或单章抖动
   - `last_action` 更新为 `{type: "write", target: "第XXX章", timestamp: 当前时间}`
   - `next_chapter_note`：如果本章写作时消费了该备注，**清空此字段**（设为空字符串或删除），防止备注在后续章节重复出现

**P1 延迟同步（不要每次自动执行，放在建议引擎中让用户决定）：**
- 如果角色状态有重大变化或有新伏笔，不要立刻消耗大量 Token 去读写档案。
- 改为在"可直接说"的建议列表中提示：`建议同步角色档案（[A]有重大变化）` 或 `建议同步伏笔表`。
- 只有当用户明确同意，或累计了多章未同步时，才集中执行 P1/P2 更新。

**P2 可选（不强制每章，每 5 章或用户主动触发）：**
4. 时间线更新 → `timeline.md`（格式参考 `Read ${CLAUDE_SKILL_DIR}/references/timeline-template.md`）
5. 新角色登场 → 创建 `characters/[新角色].md`，保存前必须先净化角色名：
   ```bash
   safe_character=$(bash ${CLAUDE_SKILL_DIR}/scripts/sanitize-filename.sh "原始角色名")
   ```

**保存完成后，显示简报：**

```
第X章「章节名」已保存 ✓
━━━━━━━━━━━━━━━━━━━━
进度：当前第X章 / 计划共Y章 / 现存Z章 | 本章约XXXX字
已更新：章节日志、项目进度
下一章：「章节名」— 大纲概要
可直接说：（按「智能建议引擎」生成上下文感知建议，含 P1 同步提醒）
━━━━━━━━━━━━━━━━━━━━
```

## 灵感激发（卡文急救）

触发：`脑暴`、`卡文了`、`灵感`、`怎么写`、`写不下去`、`下一步怎么办`

当作者遇到创作瓶颈、不知道剧情如何推进时，提供多种可能的情节走向供选择。

**上下文加载**：按第五节「智能上下文管理」的分层策略加载（L0-L5），重点关注角色动机与待揭伏笔。

**输出格式：**

根据当前剧情上下文，生成 **3-5 条差异化的情节走向建议**，每条包含：

```
《书名》灵感激发
━━━━━━━━━━━━━━━━━━━━
当前卡点：第X章「章节名」
已有线索：活跃伏笔N条、未解冲突N个
━━━━━━━━━━━━━━━━━━━━

方案一：「标题」（标签：如 爽燃/反转/伏笔回收）
  走向：一两句话描述情节发展方向
  张力来源：核心冲突/悬念是什么
  后续影响：对后文大纲的影响评估

方案二：「标题」（标签）
  ...

方案三：「标题」（标签）
  ...

━━━━━━━━━━━━━━━━━━━━
选择方案编号，或描述你自己的想法，我来帮你扩展。
```

**生成原则：**
- **差异化**：每条建议走不同方向（如：升级/反转/情感/伏笔回收/新角色登场），不要同质化
- **锚定角色动机**：建议必须符合角色已有性格和动机，不可让角色 OOC
- **伏笔优先**：优先考虑回收现有伏笔作为推进手段，避免伏笔烂尾
- **大纲兼容**：标注每条建议与已有大纲的兼容程度（完全吻合 / 小幅偏移 / 需调整大纲）
- **可执行**：建议必须具体到可以直接开写，不能是模糊的方向性描述

**用户选择后：**
- 如果用户选了某条方案 → 直接进入写作流程，以该方案为本章方向
- 如果用户提出自己的想法 → 基于用户想法进一步扩展细化，再确认后进入写作
- 如果用户只是想看看 → 不写正文，仅保留建议供参考
