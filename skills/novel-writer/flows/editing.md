# 编辑与发布流程

> 本文件包含：润色、格式整合、一致性巡检、设定编辑、角色关系图谱
> 上下文加载：一致性巡检使用**分治加载**（见巡检流程详细说明）；关系图谱按角色数量分层加载。

---

## 润色

触发：`润色`、`打磨`、`优化`、`优化文风`

**文风选项：**
- 爽文流（简洁干脆、密集金句）
- 细腻流（心理深入、意象丰富）
- 古风流（文言融入、意境古典）
- 轻小说流（口语活泼、吐槽梗）
- 悬疑流（氛围压抑、悬念感强）
- 硬科幻流（术语精准、冷峻克制）

**备份与更新机制**：
- 润色前：使用脚本归档原文件
  ```bash
  node ${CLAUDE_SKILL_DIR}/scripts/archive.js <章节文件> polish <chapters_dir>
  ```
- 润色后更新原文件
- **轻量保存**：更新 `PROJECT.yaml` 的 `updated` 字段；如果当前 `status` 为 `writing` 则更新为 `polishing`；`last_action` 更新为 `{type: "polish", target: "第XXX章", timestamp: 当前时间}`；在 `chapter-log.md` 对应章节条目的 `**字数**` 行之后追加一行 `- **润色**：YYYY-MM-DD`。如需兼容旧工作流，可选择性维护 `workspace.yaml.active_project`，但它不应覆盖当前会话里已明确的项目

## 格式整合

触发：`整合`、`导出`、`发布`、`排版`

**严禁使用大模型直接读取所有章节并输出全本**，这会必定触发输出 Token 上限导致文件截断。

**整合流程：**

1. 让大模型生成发布元信息（如书名、简介、标签、卷首语等），写入 `发布版.md` 开头，**末尾必须包含分隔符行 `<!-- CHAPTERS_START -->`**（脚本依赖此标记实现幂等，重复导出不会重复拼接）。
2. 使用专用脚本拼接所有章节正文，自动追加到分隔符之后：
   ```bash
   node ${CLAUDE_SKILL_DIR}/scripts/compile.js <项目目录路径>
   ```
3. 检查最终文件的字数与大小。

## 一致性巡检

触发：`设定检查`、`矛盾检查`、`连贯性`

**严禁全量加载所有文件**——长篇项目会直接爆 Token 上限。采用**分治巡检策略**：

**巡检流程（分治法）：**

1. **快速状态扫描**：先运行脚本获取项目概况
   ```bash
   bash ${CLAUDE_SKILL_DIR}/scripts/check-status.sh <项目目录>
   ```
2. **加载核心索引**：PROJECT.yaml + 最近 5 章日志（`tail-log.sh`）+ foreshadowing.md + timeline.md
3. **提取巡检范围**：从最近 5 章日志中提取出场角色名和涉及的地点/势力关键词
4. **定向加载**：根据提取的关键词，**只读取相关的**角色档案和世界观文件，不读取全部。大型项目（角色 > 15 或章节 > 30）可先用 `node ${CLAUDE_SKILL_DIR}/scripts/search-context.js <项目目录> "关键词" 10` 定位相关段落再精确读取
5. **执行检查**：
   - 世界观规则自洽性（仅检查被引用的设定文件）
   - 角色性格/能力一致性（仅检查出场角色）
   - 时间线逻辑（读 timeline.md）
   - 伏笔追踪（过期未揭伏笔预警）
   - 最新章节与已有设定的冲突检测
6. **大范围巡检**（仅在用户明确要求"全面巡检"或"全书检查"时）：
   - 分批处理：每次加载 10 章日志 + 相关角色/设定，检查完毕后输出中间报告
   - 批次之间通过中间报告传递已发现的问题，避免重复加载

输出检查报告 + 修复可修复的问题。

**自动修复边界（严格遵守）：**
- **可自动修复**（元数据文件）：chapter-log.md 条目补全/编号修正、foreshadowing.md 状态标记更新、timeline.md 时间格式统一、relationships.md 过期标记
- **需用户确认**（内容文件）：正文修改、角色档案修改、世界观设定修改。这些必须列入「需用户确认的修改建议」清单，由用户逐条审批后才执行

**参考模板**（按需读取）：`Read ${CLAUDE_SKILL_DIR}/references/continuity-template.md`

**巡检完成后：**
- 在 `chapter-log.md` 末尾追加巡检记录：`## [巡检] YYYY-MM-DD — 截至第X章`，后附发现的问题摘要
- 更新 `PROJECT.yaml`：`updated` 字段；`last_action` 更新为 `{type: "inspect", target: "截至第X章", timestamp: 当前时间}`。如需兼容旧工作流，可选择性维护 `workspace.yaml.active_project`，但它不应覆盖当前会话里已明确的项目

## 设定编辑

触发：`改设定`、`修改角色`、`修改世界观`、`加角色`、`新角色`

直接操作项目内的设定文件：
- 修改已有角色档案：读取 → 展示当前内容 → 按用户要求修改 → 保存
- 修改世界观设定：同上
- 新增角色档案：先净化角色名，再创建 `characters/[safe_name].md` → 更新 `PROJECT.yaml` 的 `active_characters`
  ```bash
  safe_name=$(bash ${CLAUDE_SKILL_DIR}/scripts/sanitize-filename.sh "原始角色名")
  ```
- 新增世界观文件：先净化设定名，再创建 `worldbuilding/[safe_name].md`
  ```bash
  safe_name=$(bash ${CLAUDE_SKILL_DIR}/scripts/sanitize-filename.sh "原始设定名")
  ```

**保存后更新**：
- `PROJECT.yaml`：`updated` 字段更新日期；`last_action` 更新为 `{type: "edit_setting", target: "角色/世界观文件名", timestamp: 当前时间}`；如涉及角色变动，同步 `active_characters`
- 如需兼容旧工作流，可选择性维护 `workspace.yaml.active_project`，但它不应覆盖当前会话里已明确的项目

**展示格式：**

```
设定编辑完成 ✓
━━━━━━━━━━━━━━━━━━━━
操作：修改角色「苏然」/ 新增角色「陈老」/ 修改世界观「力量体系」
变更摘要：...
━━━━━━━━━━━━━━━━━━━━
建议操作：继续写 / 一致性巡检
```

## 角色关系图谱

触发：`关系图谱`、`人物关系`、`关系网`、`谁和谁`、`角色关系`

从所有角色档案中提取关系信息，生成全局关系汇总视图，保存到 `relationships.md`。

**适用场景：**
- 人物超过 10 个，散落的关系难以掌控
- 写新章节前快速了解各角色间的关系全貌
- 检查是否有遗忘的关系线可以利用

**流程：**

1. 读取 `characters/` 目录下角色档案：
   - **禁止**默认整批读取完整角色档案；优先少读，再按需补读
   - **推荐顺序**：先读取 `PROJECT.yaml` 的 `active_characters`，再只抽取其他角色档案中的 `## 关系网` / `## 成长记录` 小节；只有在关系冲突、信息缺失或用户明确要求时，才读取完整角色档案
   - **防爆规则**：
     - 角色 ≤ 15 个：仍优先按 `active_characters` + 结构段抽取；不要默认 `cat characters/*.md`
     - 角色 16-30 个：先处理活跃角色，再按关系链补读直接相关角色档案
     - 角色 > 30 个：先用 `node ${CLAUDE_SKILL_DIR}/scripts/search-context.js <项目目录> "角色名 关系" 10` 提取相关度最高的设定片段，再按需读取完整档案
2. 提取每个角色的「关系网」和「成长记录」中的关系变化
3. 交叉比对：A→B 和 B→A 的关系描述是否一致，标注不一致的条目
4. 生成/更新 `relationships.md`
5. 更新 `PROJECT.yaml`：`updated` 字段、`last_action: {type: "relationships", target: "关系图谱", timestamp: 当前时间}`

**输出格式（relationships.md）：**

```markdown
# 角色关系图谱

> 自动生成于 YYYY-MM-DD，基于 characters/ 下所有角色档案。

## 关系矩阵

| | 苏然 | 索菲 | 陈老 | 暗影 |
|---|---|---|---|---|
| **苏然** | — | 信任/暗生情愫 | 师徒 | 宿敌 |
| **索菲** | 信任/好感 | — | 敬重 | 未知 |
| **陈老** | 师徒/器重 | 欣赏 | — | 旧怨 |
| **暗影** | 仇恨 | 未接触 | 旧怨 | — |

## 关系分组

### 核心关系（影响主线）
- **苏然 ↔ 索菲**：信任逐步建立（第3章初识 → 第7章并肩作战）
- **苏然 ↔ 暗影**：宿敌对立（第1章伏笔 → 第12章正面冲突）

### 阵营划分
- **主角阵营**：苏然、索菲、陈老
- **对立阵营**：暗影、[其他反派]
- **中立/摇摆**：[待定角色]

### 潜在关系线（尚未展开，可利用）
- 陈老 与 暗影 的旧怨（已埋伏笔，未揭）
- 索菲 与 暗影 尚无直接接触（可作为未来冲突点）

## 关系变化时间线
| 章节 | 变化 |
|------|------|
| 第3章 | 苏然 与 索菲 初识，关系：陌生→初步信任 |
| 第7章 | 苏然 与 索菲 并肩作战，关系：信任加深 |
| 第12章 | 苏然 与 暗影 正面冲突，关系：敌对升级 |

## 一致性检查
- [OK] 苏然→索菲 与 索菲→苏然 描述一致
- [!] 陈老→暗影 描述为"旧怨"，但暗影档案中未提及陈老 → 建议补充
```

**关系矩阵规则：**
- 角色少于 8 个：使用完整矩阵
- 角色 8-15 个：按阵营分组展示，只展示有实际关系的条目
- 角色超过 15 个：仅展示核心关系 + 按阵营分组，省略无交集的配对

**与其他功能的联动：**
- **章节保存**：当 P1 更新角色关系时，标记 `relationships.md` 为过期（在文件头部注明"最近更新于第X章，此后有N章未同步"）
- **一致性巡检**：巡检时自动检查 relationships.md 是否过期，若过期则重新生成
- **大纲撰写**：大纲完成后，如果已创建角色档案，自动生成初始关系图谱
